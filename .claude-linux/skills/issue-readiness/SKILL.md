GitHub Issue やタスク仕様が「無人エージェント実行（ループ）に投げられる精査度」に達しているかを判定し、欠落をユーザーへの質問として顕在化させ、回答を Issue に書き戻す。

## なぜこの skill があるか

雑な仕様でエージェントを走らせると、欠落をモデルが prior（ありがちな解釈）で補完し、意図とずれた成果物ができる。
何が欠けているかを人間が先回りして考えるのは重いので、**欠落の発見そのものをエージェントにやらせる**（質問ゼロテスト）。

精査済みの定義: 「初見の実装者が、追加質問なしに、受け入れられる成果物を出せる状態」（cf. スクラムの Definition of Ready）。
「ゼロ」は全質問ではなく**意図レベルの質問がゼロ**という意味。設計・機械レベルの曖昧さは停止規則や完了条件に変換してループに吸収できるが、意図の取り違えだけは成果物を丸ごと無駄にする。
背景の整理は work-vault の [[loop-engineering]] を参照。

## 入力

以下のいずれか。指定がなければユーザーに確認する。

- GitHub Issue（URL または `owner/repo#番号`）→ `gh issue view` で title/body/labels/comments を取得
- ファイルパス（仕様メモ・Issue 草案）
- 会話に貼られたテキスト
- 「これから Issue を書く」という相談 → 口頭の依頼内容を草案とみなして同じ手順を回す

## 手順

### 1. 自答フェーズ — 資本から答えられる質問を消す

質問を列挙する前に、プロジェクト側の知識で答えが出るものを**出典付きで自答**し、ユーザーへの質問から除外する。規約に書いてあることを聞き返すと、本当に重要な質問が埋もれてユーザーの注意を浪費する。参照先:

- 対象リポジトリの CLAUDE.md・README・規約
- Issue 本文からリンクされたドキュメント
- 類似の過去 Issue / PR（`gh search issues` / `gh pr list`）
- 既存コードの慣習（Grep で確認）

### 2. 仮定と質問の列挙

「このまま実装に着手した場合に置くことになる仮定」を全部挙げ、3レベルに分類する:

- **A. 意図レベル** — 何を作るか・どこまでやるか・どちらを優先するか。間違うと成果物が丸ごと無駄になる
- **B. 設計レベル** — 作り方・構成・流儀。規約や既存コードで足りるなら質問にしない。残っても「迷ったら実装せず止まって報告」という停止規則に変換できる
- **C. 機械レベル** — テスト・命名などの些末。質問にせず完了条件に落とす

各項目に「**答えが変わると成果物がどう変わるか**」を1行添える。成果物が変わらない質問は捨てる。質問数を稼ぐことが目的ではない — 本質的な質問1件は網羅的な質問10件より価値がある。

分類例 — 「アクセスログを集計するスクリプトが欲しい」という Issue:

| レベル | 仮定/質問 | 扱い |
|---|---|---|
| A | 集計単位は時間別か、ステータス別か、URL別か（成果物の根本が変わる） | ユーザーに質問 |
| B | 実装言語（リポジトリの慣習が shell なら自答できる） | 出典付き自答 or 停止規則 |
| C | 出力ファイル名・配置先 | 完了条件に落とす |

### 3. 判定とレポート

- **NOT READY**: A が1件でも残っている
- **READY**: A が空。READY でも暗黙の前提は【推測】【未検証】マークで明示する

以下の形式で提示する:

```markdown
## Readiness 判定: READY / NOT READY

### 自答済み（質問から除外）
- Q: ... → A: ...（出典: <file/PR>）

### A. 意図レベル — 要回答（影響が大きい順）
1. <質問>（答えが変わると: <成果物への影響>）

### B/C — 停止規則・完了条件へ変換
- <内容> → <変換先>

### 暗黙の前提
- 【推測】<内容>
```

**ユーザーに質問できない文脈**（サブエージェント・自動ループ・`claude -p` からの起動）では、ここで終了してレポートを最終出力として返す。Issue への書き込みは行わない。

### 4. ユーザーへの質問

NOT READY の場合、A の項目を影響度順に最大5件、AskUserQuestion で確認する（選択肢を作れるものは選択式に。「その他」で自由回答できるので選択肢の網羅は不要）。回答が新たな分岐を生んだら 2〜3 を繰り返す。

### 5. 書き戻し

回答を反映した Issue 本文を以下の構成で提案し、**確認を取ってから**反映する（`gh issue edit` またはファイル編集）:

```markdown
## 目的（なぜ）
## 完了条件（機械検証可能なものを優先: コマンド・テスト）
## 範囲（触ってよい / 触ってはいけない）
## 前提（出典付き。出典がないものは【推測】【未検証】）
## 分岐への先回り回答・停止規則
```

READY になったら `loop-ok` ラベルの付与を提案する。

## 禁止事項

- 実装に着手しない（readiness 判定のみで終える）
- ユーザーの代わりに意図レベルの答えを創作しない
- 出典のない前提を事実として書き戻さない
