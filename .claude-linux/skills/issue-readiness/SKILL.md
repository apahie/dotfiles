# issue-readiness — 質問ゼロテスト

Issue やタスク仕様の「精査度」を判定する。精査済みの定義: **初見の実装者（人でもモデルでも）が、追加質問なしに、依頼者が受け入れる成果物を出せる状態**（スクラムの Definition of Ready に相当）。

仕様に欠落があると、実装エージェントはそれを prior（ありがちな解釈）で埋め、意図とずれた成果物を作る。欠落を依頼者が先回りして洗い出すのは重いので、**欠落の発見そのものをエージェントにやらせる**のがこのスキル。「ゼロ」の意味は質問が文字通りゼロではなく、**意図レベルの質問がゼロ**になったら合格。

## 原則: 分析は文脈を持たないサブエージェントで行う

判定したいのは「初見の実装者が質問なしで作れるか」。メインセッションは会話の中で依頼者の意図を既に知っていることが多く、冷たい読者を演じられない（既に知っていることは質問として出てこない）。**会話文脈を共有しないサブエージェントこそが、初見の実装者の忠実なシミュレーション**。この構成はどのセッション・どの環境から呼ばれても同じ品質で動く。

- サブエージェント（Task/Agent ツール）が使える場合: 必ずサブエージェントで分析する。読み取り中心のタスクなので read-only 系のエージェントタイプで十分
- 使えない場合のみインラインで分析する。その際、会話で得た知識を根拠にしてはならない。Issue 本文と参照可能なファイルだけを根拠とし、会話で知った意図はむしろ「仕様に書かれていない＝欠落」として報告する

## ワークフロー

### 1. 入力の実体化

対象は GitHub Issue（URL / `owner/repo#番号`）、ファイルパス、または貼り付けテキスト。指定がなければユーザーに確認する。

サブエージェントは会話を読めないので、入力は必ずファイルか Issue 参照として渡す:

- Issue → repo と番号を渡す（サブエージェント側で `gh issue view` させる）
- 貼り付けテキスト → 一時ファイル（例: `/tmp/issue-draft.md`）に保存して渡す
- 対象リポジトリのローカルパスが分かれば渡す（自答フェーズの精度が上がる）

### 2. 分析サブエージェントの起動

後述の「分析プロンプト」の `{ }` を埋めて、そのままサブエージェントに渡す。プロンプトは自己完結しており、このスキル本文や会話の知識を前提にしない。

### 3. 結果の提示

返ってきた Readiness Report を要約せずそのまま提示する。判定（READY / NOT READY）・A レベルの質問・【推測】マークが依頼者に見えることが重要。

### 4. 質問（対話できる場合のみ）

NOT READY なら、A レベルの項目を影響度順に最大5件、AskUserQuestion で確認する（選択肢を作れるものは選択式に）。回答が新たな分岐を生んだらステップ 2 を再実行する。

**非対話実行の場合**（このスキル自体がサブエージェント・ループ・スケジュール実行の中で動いていて、ユーザーに質問できない場合）: 質問せず Readiness Report を返して終了する。このとき、回答待ち箇所を `◆[Q番号]` スロットで示した書き戻し草案をレポートに含めると、依頼者は回答を埋めるだけで済む。呼び出し元から指示されている場合のみ、質問リストを Issue コメントとして投稿する。

### 5. 書き戻し

回答を反映した Issue 本文を以下の構成で提案し、**依頼者の確認を取ってから**反映する（`gh issue edit` またはファイル編集）:

```markdown
## 目的（なぜ）
## 完了条件（機械検証可能なものを優先: コマンド・テスト）
## 範囲（触ってよい / 触ってはいけない）
## 前提（出典付き。出典がないものは【推測】【未検証】）
## 分岐への先回り回答・停止規則
```

READY になったら `loop-ok` ラベルの付与を提案する。

## 分析プロンプト（サブエージェントに渡すテンプレート）

```text
あなたは「このタスクを初めて見る実装者」です。実装には着手せず、
タスク仕様の精査度（着手準備が整っているか）を判定してください。

## 対象
- 仕様: {ファイルパス、または「Issue #N を gh issue view <N> --repo <owner/repo> --json title,body,labels,comments で取得」}
- 対象リポジトリ: {ローカルパス。不明なら「なし（仕様のみで判断）」}

## 手順
1. 自答フェーズ: 仕様を読んで湧いた疑問のうち、リポジトリ側の知識で答えが
   出るものを出典付きで自答し、質問リストから除外する。参照先:
   CLAUDE.md / README / 規約 / 仕様からリンクされた文書 /
   類似の過去 Issue・PR（gh で検索）/ 既存コードの慣習（grep で確認）
2. 「このまま実装に着手した場合に、自分が置くことになる仮定」を全部列挙し、
   3レベルに分類する:
   - A. 意図レベル: 何を作るか・どこまでやるか・どちらを優先するか
   - B. 設計レベル: 作り方・構成・流儀。規約や既存コードで決まるなら質問にしない
   - C. 機械レベル: テスト・命名などの些末。質問にせず完了条件への提案に落とす
   各項目に「答えが変わると成果物がどう変わるか」を1行添える。
   答えが変わっても成果物が変わらない質問は捨てる。
   残った欠落は「誰かに聞けば決まるもの（→質問）」と「現実を測れば
   分かるもの（→【未検証】・実測 PoC）」に振り分ける。
3. 判定: A が1件でも残れば NOT READY、空なら READY。
   READY でも暗黙に置く前提は【推測】マークで明示する。
   出典が無く実測でしか確認できない前提は【未検証】とし、
   「実装の最初のタスクとして実測 PoC で確認する」ことを提案する。
   PoC には、仕様が触る実物（ファイル・データ・API 応答など）を最低
   1つ開いて観察することを含める。環境の事実は事前の列挙では
   網羅できないため、観察で拾う。
   既存資産・前例に倣う前提を置く場合は、踏襲元の前提条件（実行環境・
   形式・頻度など）が本件に当てはまるかを確認対象に含める。
   出力前に「自答済みの事実」と「暗黙の前提」を突き合わせ、矛盾が
   あれば前提を修正するか質問に昇格する。

## 出力形式（この構成のまま返す）
# Readiness Report
- 判定: READY / NOT READY
- 対象: <仕様のタイトル>

## A. 意図レベルの質問（影響度順）
- Q1: <質問> ／ 影響: <答えで成果物がどう変わるか>

## B. 設計レベル（規約参照・停止規則で吸収可能）
## C. 機械レベル → 完了条件への提案
## 自答済みの質問（出典付き）
- Q: <質問> → A: <答え>（出典: <path:line または URL>）

## 暗黙の前提【推測】
```

## 禁止事項

- このスキルの中で実装に着手しない（readiness 判定で止める）
- 依頼者の代わりに意図レベル（A）の答えを創作しない
- 出典のない前提を事実として Issue に書き戻さない
